# FitnessClub Agent

Агент клуба: **QR → CRM (клиент + абонемент) → открытие турникета C01**.  
Все HTTP-запросы к CRM и команды на C01 пишутся в **Журнал**.

## EXE

`dist\FitnessClubAgent.exe` — закройте старую копию перед пересборкой `.\build.ps1`.

## Быстрый старт

1. **CRM** — в поле URL укажите **только корень сайта**, без `/admin` и без `/api/v1`  
   (пример: `https://worldcashfit.ru`). Иначе heartbeat уйдёт на `.../admin/api/v1/...` и вернётся **404**.  
   Дальше `gateway_token` → **Сохранить всё** → **Проверить CRM** (в журнале `HTTP 200`).
2. **Оборудование** — режим **Слушать**, host `0.0.0.0`, порт `8765`; в **net.server** укажите **`192.168.0.63:8765`**, если PERCo принимает порт в строке сервера (см. ниже). На ПК откройте **входящий TCP 8765** в брандмауэре.
3. **Запустить агент** — C01 **[ON]** (или **симулятор** без железа).
4. **Проход** — QR с телефона (< 15 сек) → **Проверить проход** — смотрите **Журнал**.

## Что в журнале при скане

```
══════ ПРОХОД: скан QR ══════
← C01 event card | id=FITNESSCLUB:ENTRY:...
────── CRM запрос ──────
→ POST /api/v1/gateway/access/entry
  тело: {"qr":"...", "device_id":"..."}
← HTTP 200
  тело: {"access_granted":true, "user":{...}}
  итог: ДОПУСК — Иван Иванов
────── C01 команда (открыть турникет) ──────
→ {"control":"exdev", ...}
```

При отказе: `access_granted:false`, `reason=...`, команда **запрет прохода**.

## Почему скан QR «ничего не делает»

1. **В шапке должно быть `C01: 1/1 подключено`.** Если `0/1` — события со считывателя до агента не доходят: сначала наладьте WebSocket (см. ниже).
2. **Скан с турникета:** агент шлёт строку QR в CRM `POST /api/v1/gateway/access/entry` — там проверяются формат, срок 15 сек, пользователь и **активный абонемент**. Ответ `access_granted: true` → команда открыть турникет; иначе → запрет. События PERCo **`card`** и **`pass_personal`** обрабатываются одинаково (часто штрихкод даёт именно `pass_personal`).
3. **Проверка без считывателя:** вкладка **«Проход»** — вставьте свежий QR из приложения (менее 15 секунд с момента показа) → **Проверить проход**. В **«Журнал»** должны появиться строки `ПРОХОД` и запрос в CRM.
4. **Без железа:** в списке оборудования можно включить **симулятор** (если есть в вашей сборке) — тогда тест прохода идёт без C01.

### В журнале только цифры (`14829991`), а не `FITNESSCLUB:ENTRY:...`

Считыватель или прошивка PERCo отдают **усечённый числовой код**, а мобильное приложение кодирует **полную ASCII-строку** с двоеточиями. CRM проверяет именно формат приложения — без полной строки абонемент не проверить.

Что сделать: в веб-интерфейсе PERCo (reader / тип сканера) включите передачу **полного содержимого QR**; не используйте режим «только номер карты» / Wiegand-ограничение по длине. Контроль: вкладка **«Проход»** в агенте — вставьте строку с экрана телефона вручную — в журнале должен появиться полный `FITNESSCLUB:...`, затем **Проверить проход** и ответ CRM.

### Что сказано в **C01_Protocol.pdf** (начало документа) — часто это и есть «упущенное»

1. **Два этапа подключения (стр. 1):**  
   **Пока** контроллеру **не задан IP «сервера системы»**, **подключение к контроллеру должен инициировать сам сервер** (ваш агент в режиме «К контроллеру») **или** используется **веб-браузер с WebSocket**.  
   **После** того как IP сервера задан (через браузер или по протоколу), **контроллер сам подключается к серверу** — тогда уместен режим **«Слушать»** на ПК.

2. **Поле `net.server` в JSON (раздел 2.1):** в примере протокола указан **только IP** (`"server": "192.168.0.250"`), **без порта**. На практике прошивка может использовать **фиксированный порт по умолчанию** (часто **80** для `ws://` или **443** для `wss://`). Если в веб-интерфейсе PERCo **нельзя** задать порт и при этом **0/1** при слушании **8765** — имеет смысл временно поставить в агенте **«Порт (для C01)» = 80**, запустить от **администратора** и освободить порт 80 на ПК (без IIS/Apache), либо уточнить порт у **soft@perco.ru**.

3. **Пароль (раздел 1):** если на контроллере включён пароль доступа, после соединения придёт `need_auth` — в агенте поле **«Пароль»** должно совпадать с **«Пароль доступа» / net.password** на C01. Пустой пароль с обеих сторон — нормально.

4. **Диагностика:** на ПК **192.168.0.63** при перезагрузке C01 посмотрите **Wireshark** (фильтр `tcp && ip.addr==192.168.0.85`) — идут ли **SYN** с `.85` на ваш порт (**8765** или **80**). Если пакетов нет — контроллер не пытается выйти на сервер (проверьте сохранение «Сервер» в веб-интерфейсе, перемычки **IP MODE / IP DEFAULT**, п. 3 «Состояние» в PDF).

### Режим «К контроллеру» и ошибка `HTTP 200` в журнале

Сообщение вида `server rejected WebSocket connection: HTTP 200` значит: по указанному адресу отвечает **обычный HTTP** (часто **порт 80** — веб-интерфейс), а не протокол WebSocket C01. Нужен **реальный порт WS** контроллера из паспорта/инструкции C01, либо другой сценарий:

### Рабочий сценарий после настройки сети (рекомендуется)

1. Режим **«Сервер системы (слушать)»**, host `0.0.0.0`, порт **`8765`**.
2. **Порт сервера на C01:** если в PERCo в «Адрес сервера» только IP без порта, контроллер часто ходит на **:80**, а агент слушает **8765** — соединения не будет. Варианты: в веб-интерфейсе PERCo задать **`192.168.0.63:8765`** (если поле поддерживает); или в мануале найти отдельное поле **порта сервера** и указать **8765**; или (крайний случай) поднять агент на порту, который ждёт прошивка.
3. Задайте сервер на **самом PERCo** (веб `192.168.0.85` → **Сервер**): **`192.168.0.63:8765`**, сохраните, перезагрузите C01 при необходимости. Кнопки **«Прочитать/Записать net»** в агенте в режиме «Слушать» смысла **до появления 1/1** не имеют (в новой версии агента вместо «HTTP 200» показывается пояснение).
4. **Брандмауэр Windows** — входящий **TCP 8765** для частной сети.
5. Перезагрузка / применение настроек на C01 по инструкции PERCo.
6. Перезапуск агента. В шапке — **`C01: 1/1 подключено`**, в журнале — `C01 подключился`.
7. **Журнал «Запрос к WS с …»:** если при сохранении настроек на PERCo или перезагрузке контроллера **нет** такой строки — с ПК **не приходит** TCP на порт агента (другой порт по умолчанию у прошивки, неверный `net.server`, VLAN, брандмауэр). Если строка **есть**, а `C01 подключился` **нет** — рукопожатие WebSocket отвергнуто (в актуальной версии агента отключено сжатие `permessage-deflate` ради совместимости с PERCo).
8. Дальше при необходимости **«Прочитать/Записать net»** в агенте — уже по установившемуся WebSocket.

## Архив **ctl_websock C01** от PERCo — что это

Типичная поставка техподдержки: **учебный Python‑клиент** к контроллеру по WebSocket и набор **готовых JSON‑команд** в папке `commands/` (`net_get.json`, `exdev_control_open.json` и т.д.).

Как пользоваться (из корня архива, где лежит `main.py`):

1. Установить зависимости: `pip install websockets` (версия из их кода может отличаться от вашей).
2. Запуск: `python main.py` — откроется консоль **`main$`**.
3. **`connect 192.168.0.85`** — клиент подключается к **`ws://192.168.0.85:80/tcp`** (по умолчанию порт **80**, путь **`/tcp`**, не корень `/`).
4. **`start 8080`** — поднять **тестовый WebSocket‑сервер** на указанном порту (в коде по умолчанию **80** без аргумента) — имитация «сервер системы», к которому может подключиться C01.
5. **`run net_get.json`** — отправить JSON из файла и дождаться ответа; **`list`** — список файлов в `commands/`.
6. **`disconnect`** / **`bye`** — остановить сессию и выйти.

Важно для **FitnessClub Agent**: в эталоне PERCo исходящий URL — с суффиксом **`/tcp`** и заголовком **`Origin`** = IP контроллера; в агенте это учтено в режиме **«К контроллеру»** (поле «Порт WS на C01» по-прежнему задаёт порт, чаще всего **80**).

## HTTP 401 на heartbeat

Тело `{"ok":false,"reason":"unauthorized"}` — в CRM **нет клуба** с таким `gateway_token`. Не путать с логином пользователя приложения: нужен именно токен из карточки клуба в админке. После ротации токена в CRM обновите его в агенте.
